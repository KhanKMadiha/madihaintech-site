What this project is

This is a live, production AI chatbot embedded across my portfolio site. It answers questions about my experience, skills, and career goals on behalf of me, helping recruiters explore my background interactively rather than just reading static text.

What started as a simple Dialogflow CX experiment became a full journey through API integration, security challenges, and infrastructure decisions. The chatbot you see in the bottom right corner today is the result of that entire process.

The full journey

  • Phase 1

    Starting with IBM Watson and Dialogflow CX

    Learning the fundamentals of conversational AI

    I started by completing the IBM AI Fundamentals certification and building a prototype using IBM Watson to understand intents, entities, and dialogue flows. I then moved to Google Dialogflow CX, which gave me more control over multi-turn conversations and intent routing.

    I embedded the Dialogflow bot into this portfolio and it worked, until it didn't. Testing it regularly, I kept hitting a Permission Denied error that I couldn't resolve. The root cause came down to domain allowlisting and integration settings inside the Dialogflow console, but after spending time investigating it, I made the decision to move on rather than continue troubleshooting a tool that wasn't serving me well.

    Struggle: The Dialogflow CX embed required the Messenger integration to be explicitly enabled and domains allowlisted inside the Google Cloud console. Without this, every request returned a permission denied error regardless of the code being correct. Debugging this across Google Cloud IAM, Dialogflow settings, and the front-end embed was time consuming and not well documented.
  • Phase 2

    Switching to the Anthropic API

    Replacing intent-based flows with a language model

    Rather than continuing to map every possible question to a predefined intent, I switched to using Claude via the Anthropic API directly. This meant writing a detailed system prompt describing my background, experience, and tone, and letting the model handle any question a recruiter might ask naturally.

    The difference was immediate. No more intent mapping, no training data, no permission configuration. I built a custom chat widget in HTML and JavaScript that matched my existing pink theme, with a proactive welcome message that opens automatically two seconds after the page loads.

    Struggle: The first version had the Anthropic API key hardcoded directly in the front-end JavaScript. GitHub's secret scanning flagged it immediately on push and blocked the commit, a valuable lesson in API key security and the importance of never committing secrets to a public repository.
  • Phase 3

    Securing the API key with Cloudflare Workers

    Building a proxy to keep secrets off the front end

    To fix the security issue properly, I set up a Cloudflare Worker as a proxy. The Worker sits between my portfolio site and the Anthropic API. My front end sends requests to the Worker, the Worker adds the API key server-side, and forwards the request to Anthropic. The key never appears in my code or in the browser.

    I configured the Worker to only accept requests from madihaintech.me using CORS headers, stored the API key as an encrypted environment secret in Cloudflare, and deployed it on the free tier with no credit card required.

    Struggle: Setting up the Cloudflare Worker involved navigating an unfamiliar console, figuring out how to add encrypted secrets, and understanding CORS preflight requests. The Cloudflare editor also did not support pasting on the initial deploy screen, which required deploying a Hello World worker first and then replacing the code afterwards in the full editor.
  • Phase 4

    Live and iterating

    Production deployment across the full portfolio

    The chatbot is now live across all pages of this portfolio, powered by Claude, secured via Cloudflare, and deployed through GitHub Pages. It opens automatically after two seconds, delivers a structured welcome message, and handles follow-up questions with full conversation history maintained client-side for the session.

    The system prompt is carefully designed to keep responses warm, professional, and grounded only in accurate information about my background, with a fallback to direct visitors to my contact page or LinkedIn if something falls outside what the assistant knows.

What I learned

This project taught me more than any tutorial could. I learned that real AI integration involves infrastructure decisions, not just model prompts. Where your API key lives, how requests are routed, and how you handle errors all matter as much as what the AI says.

I also learned that iteration beats perfection. The Dialogflow version was imperfect but it shipped. The permission error pushed me toward a better solution. The exposed API key forced me to understand proxies and secrets management. Every struggle led somewhere useful.
